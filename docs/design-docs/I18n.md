# I18n

Ankole I18n loads application-owned translation catalogs from TOML files and
resolves translations through `Ankole.I18n`.

The implementation lives in `Ankole.I18n.*`, `AnkoleWeb.I18n.*`, Phoenix
helpers, and the AppConfigure definition that owns the operator-selected
default locale.

## Catalog Files

Catalogs live under the app-level shared root, outside Phoenix `priv`, so the
control plane and browser apps can read from the same source directory:

- `app/locales/en-US.toml`
- `app/locales/zh-Hans-CN.toml`

The filename without `.toml` is the locale id. Locale ids are stored and passed
as strings, not atoms, so request and configuration input never creates atoms at
runtime.

Client-only catalogs or generated browser projections may live under the
parallel browser surface when a JavaScript client needs strings that should not
be loaded by the server runtime:

- `app/locales/client/en-US.toml`
- `app/locales/client/zh-Hans-CN.toml`

Server and client catalogs are application/deployment assets. They are not
AppConfigure values and are not editable through runtime configuration.

## Configuration

I18n has one runtime configuration value:

- `i18n.default_locale`, default `en-US`.

The key is declared through `Ankole.AppConfigure` as a non-empty locale id
string. It is plaintext, scoped like any other AppConfigure key, and resolves
through the normal AppConfigure order: agent scope, global scope, then the code
default.

AppConfigure stores the operator's locale choice. The catalog runtime decides
whether that locale is currently available. `Ankole.I18n.put_default_locale/1`
validates the requested locale against the loaded catalog before persisting, and
catalog startup/reload validates the effective stored value before applying it to
Localize.

Catalog paths are bootstrap/deployment facts, not operator runtime settings.
The control-plane runtime defaults to `../locales` from `app/control_plane`,
which resolves to `app/locales` in the source tree. A deployment that packages
the shared directory differently must override `config :ankole, :i18n,
locales_dir: ...`. Changing catalog files or their directory is a code or
deployment change.

## Server Catalog Runtime

`Ankole.I18n.Catalog` is a GenServer under `Ankole.Application`. It starts after
`Ankole.AppConfigure.Cache` and before subsystems that render text. On startup
it:

1. registers I18n's AppConfigure definition;
2. scans `app/locales/*.toml`;
3. validates that the effective AppConfigure default locale is loaded;
4. updates Localize supported locales and default locale;
5. writes messages and metadata into `:persistent_term` through
   `Ankole.I18n.Resolver`.

`Ankole.I18n.reload/0` re-reads the effective AppConfigure default locale.
`Ankole.I18n.reload_locales/0` re-scans the application catalog files and then
re-applies the effective default locale.

Runtime default-locale changes should go through `Ankole.I18n.put_default_locale/1`.
That write path persists the AppConfigure value and reloads the Localize default
for the current node. Direct AppConfigure writes remain valid storage writes,
but they do not imply catalog/runtime projection hooks by themselves.

## TOML Shape

Each locale file may include `__meta__`:

```toml
[__meta__]
bcp47 = "zh-Hans-CN"
fallback = "en-US"
```

Nested TOML tables become dotted translation keys. String leaves are MF2 message
sources. Rich leaves may use this shape:

```toml
[example.cta]
__mf2__ = true
message = "Open {#strong}Ankole{/strong}"
description = "Optional translator note"
```

Only the `message` field is loaded at runtime. Descriptions and placeholders are
translator metadata.

## Fallback Chain

`Ankole.I18n.Resolver` tries translations in this order:

1. requested locale;
2. the requested locale's `__meta__.fallback`, when present;
3. application default locale;
4. `en-US`.

Only loaded full tags are used. Ankole does not infer parent tags by trimming
BCP 47 subtags.

## Server API

`Ankole.I18n.t/3` returns a string and falls back to a literal key marker when no
translation is found.

`Ankole.I18n.translate/3` returns structured success or error tuples.

Other current calls include:

- `default_locale/0`
- `put_default_locale/1`
- `get_locale/0`
- `put_locale/1`
- `with_locale/2`
- `reload/0`
- `reload_locales/0`
- `available_locales/0`

Callers may pass `scope: "some.scope"` to compose local keys, but shared server
and client catalogs should prefer explicit dotted keys.

## Phoenix Integration

Phoenix controllers, components, and templates import `AnkoleWeb.I18n.HTML`.
The root layout uses the active default locale for `<html lang>` and a single
`dir/0` helper as the future RTL hook.

Changeset error translation is implemented under `AnkoleWeb.I18n` and maps Ecto
validation metadata onto stable translation keys before calling `Ankole.I18n`.

Setup and console surfaces should change the default locale through
`Ankole.I18n.put_default_locale/1`, not by writing undeclared config keys.

## Client Runtime

Browser clients should consume `app/locales/client/*.toml` or a generated
projection under `app/locales` when they need client-only strings. Shared strings
belong at the catalog root so the server and browser surfaces do not drift.

## Validation

Catalog loading validates TOML shape and MF2 sources. A dedicated Mix validation
task may additionally enforce cross-locale key drift before release.

Tests cover loader, resolver, catalog, server API, Phoenix error translation,
and AppConfigure default-locale integration.

## Invariants

- Translation catalogs do not store secrets, tokens, activation codes, login
  codes, provider raw payloads, or private user data.
- Locale ids are full tags such as `en-US` and `zh-Hans-CN`.
- The default locale is AppConfigure key `i18n.default_locale`; catalog paths
  and catalog contents are not AppConfigure values.
- Missing translations are visible failures, not silent blank strings.
- I18n only renders text; authorization and business policy decisions happen in
  their owning subsystems before rendering.
