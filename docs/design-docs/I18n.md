# I18n

BullX I18n is the cross-cutting translation and localization subsystem. It
uses stable dotted keys, TOML catalog files, and MessageFormat 2 (MF2) message
formatting so server code, Phoenix views, channel adapters, and React pages can
render localized human-facing text without coupling source strings to code.

The current design keeps locale selection application-global. One BullX
Installation has one active runtime locale, resolved through `BullX.Config`.
BullX does not negotiate locale per browser, per Principal, per Connected Realm,
or per channel actor.

## Scope

This design covers:

- server-side translation through `BullX.I18n`;
- TOML server catalogs under `priv/locales/*.toml`;
- Phoenix helpers under `BullXWeb.I18n`;
- React translation through `webui/src/i18n`;
- TOML client catalogs under `priv/locales/client/*.toml`;
- locale configuration through `BullX.Config.I18n`;
- offline catalog validation through `mix i18n.check`.

The subsystem owns translation lookup and locale formatting. It does not own
which product copy a feature should show, which channel messages are sent, or
whether an external side effect is authorized.

## Non-goals

The first I18n design intentionally excludes:

- per-user or per-Principal locale preferences;
- `Accept-Language` negotiation;
- URL locale scopes such as `/:locale/...`;
- browser language detection or `localStorage` locale state;
- runtime translation catalog export endpoints;
- Inertia props that carry complete translation catalogs;
- a shared runtime engine between Elixir `Localize` and browser
  `messageformat`;
- Phoenix Gettext catalogs or `.po` files;
- JSON or YAML translation catalogs.

## Existing system

The current implementation is already in tree:

- `BullX.I18n` is the public server API.
- `BullX.I18n.Catalog` loads server catalogs at boot and stores normalized
  catalogs in `:persistent_term`.
- `BullX.I18n.Loader` scans `priv/locales/*.toml` non-recursively.
- `BullX.I18n.Normalizer` flattens TOML tables, validates MF2, and stores
  canonical message sources.
- `BullX.I18n.Resolver` builds fallback chains and resolves keys.
- `BullX.Config.I18n` declares `bullx.i18n_default_locale` and
  `bullx.i18n.locales_dir`.
- `BullXWeb.I18n.HTML` provides Phoenix template helpers and root `<html>`
  language attributes.
- `BullXWeb.I18n.ErrorTranslator` maps Ecto changeset errors to deterministic
  translation keys.
- `webui/src/i18n` initializes React `i18next` with an MF2 post-processor.
- `rsbuild.config.ts` imports client TOML files through the local TOML loader
  plugin and watches `priv/locales/client/*.toml` in development.
- `Mix.Tasks.I18n.Check` validates server and client catalog drift.

`Localize` is the server MF2 parser and formatter. BullX uses `Localize` for
MF2 parsing, CLDR-backed formatting, locale tags, and process-local formatting
state. BullX owns key lookup, catalog loading, catalog fallback, and catalog
validation.

## Locale model

The active locale is an Installation-wide runtime setting:

| Concern | Owner | Shape |
| --- | --- | --- |
| Default locale | `BullX.Config.I18n.i18n_default_locale!/0` | BCP 47 string, default `"en-US"` |
| Server catalog directory | `BullX.Config.I18n.i18n_locales_dir!/0` | Path string, default `"priv/locales"` |
| Loaded locale ids | `BullX.I18n.Catalog` and `BullX.I18n.Resolver` | Atom ids derived from TOML filenames |
| Phoenix `<html lang>` | `BullXWeb.I18n.HTML.lang/0` | BCP 47 string from the loaded default locale |
| Phoenix `<html dir>` | `BullXWeb.I18n.HTML.dir/0` | `"ltr"` until BullX adds an RTL locale |
| React active locale | `webui/src/i18n/i18n.ts` | `document.documentElement.lang`, falling back to `"en-US"` |

Locale changes go through `BullX.Config.put/2` or the corresponding runtime
configuration source. `BullX.I18n.reload/0` applies a changed default locale
without rescanning catalog files. `BullX.I18n.Catalog.reload_locales!/0`
rescans files and is used by tests and the development watcher.

`BullX.I18n.put_locale/1` and `BullX.I18n.with_locale/2` remain available for
tests, debugging, and offline rendering. Normal web and adapter code should call
`BullX.I18n.t/3` or `translate/3` and let the global locale select the catalog.

## Server catalogs

Server catalogs live in `priv/locales/*.toml`. The loader reads only TOML files
directly under the configured directory and ignores subdirectories such as
`priv/locales/client`.

Each server filename is the loaded locale id:

```text
priv/locales/en-US.toml       -> :"en-US"
priv/locales/zh-Hans-CN.toml  -> :"zh-Hans-CN"
```

Every locale file uses a nested TOML table tree. The normalized message key is
the dotted path from the root table to a string leaf:

```toml
[eventbus.feishu.auth]
activation_success = "Activation complete."
```

The preceding example becomes the key
`eventbus.feishu.auth.activation_success`.

`__meta__` is reserved for catalog metadata and does not become part of the key
tree:

```toml
[__meta__]
bcp47 = "zh-Hans-CN"
fallback = "en-US"
revision = "2026-05-19"
```

Allowed metadata keys are `bcp47`, `fallback`, and `revision`. Metadata values
must be strings.

Catalog leaves must be either a string MF2 source or a rich-leaf table:

```toml
[app.button]
save = "Save"

[app.checkout.total]
__mf2__ = true
message = "Checkout {$total :currency currency=$currency}"
description = "Checkout button total."
placeholders = { total = "decimal amount", currency = "ISO 4217 code" }
```

`BullX.I18n.Normalizer` stores only the MF2 message source in the runtime
catalog. `description` and `placeholders` are authoring metadata for translators
and review tooling.

## Fallback chain

`BullX.I18n.Resolver` resolves a key by trying a deduplicated locale chain in
this order:

1. The requested locale.
2. The requested locale's `__meta__.fallback`, when it names a loaded locale.
3. The application default locale.
4. `:"en-US"`.

Every chain entry must be a loaded locale. BullX does not synthesize parent
locales by truncating BCP 47 tags. For example, requesting `zh-Hans-CN` does not
automatically try `zh-Hans` or `zh`. A deployment that wants that behavior must
ship a `zh-Hans.toml` file and reference it through `__meta__.fallback` or make
it the default locale.

Fallback is a translation lookup behavior, not a locale negotiation behavior.
The active application locale remains the configured default unless a caller
explicitly asks `BullX.I18n` to format with another loaded locale.

## Server API

`BullX.I18n` exposes the server-side public API:

```elixir
BullX.I18n.t("eventbus.feishu.auth.activation_success")
BullX.I18n.t("examples.greeting", %{name: "Alice"}, locale: :"zh-Hans-CN")
BullX.I18n.translate("examples.greeting", %{name: "Alice"})
BullX.I18n.available_locales()
BullX.I18n.reload()
```

`t/3` returns a string. Missing keys return the key literal and log an I18n
missing event. MF2 formatting failures return the raw MF2 source and log an I18n
format error.

`translate/3` returns `{:ok, string}` or `{:error, exception}` without logging a
missing-key or formatting error. External side-effect boundaries should use
`translate/3` when sending a key literal would be worse than not sending a
message.

The second argument is MF2 bindings. The third argument is I18n options. This
keeps reserved I18n options such as `:locale` and `:scope` out of MF2 bindings:

```elixir
BullX.I18n.t("greeting", %{name: "Alice"}, scope: "examples")
```

`use BullX.I18n, scope: "some.scope"` injects scoped `t/1`, `t/2`, and `t/3`
helpers for modules that own a stable key prefix. Scoped keys are still normal
catalog keys after concatenation.

## Phoenix integration

Phoenix-specific behavior stays under `BullXWeb.I18n`.

`BullXWeb.I18n.HTML` is imported into modules that use the `BullXWeb.html/0`
surface. It delegates directly to `BullX.I18n.t/3`; it does not keep a Phoenix
locale assign, session value, or Plug.

`BullXWeb.I18n.HTML.lang/0` and `dir/0` feed the root layout:

```heex
<html lang={BullXWeb.I18n.HTML.lang()} dir={BullXWeb.I18n.HTML.dir()}>
```

`BullXWeb.I18n.ErrorTranslator` converts Ecto error tuples into deterministic
keys. It prefers structured validation metadata, then falls back to a normalized
message key:

```text
validation: :length, kind: :min, type: :string
-> errors.validation.length.string.min

validation: :required
-> errors.validation.required

"can't be blank"
-> errors.cant_be_blank
```

Remaining Ecto error options are passed as MF2 bindings.

## Client catalogs

React translations live under `priv/locales/client/*.toml`. Client catalogs use
the same TOML format and BCP 47 filename convention as server catalogs, but they
are a separate runtime:

| Catalog | Loader | Runtime |
| --- | --- | --- |
| `priv/locales/*.toml` | `BullX.I18n.Catalog` | Elixir `Localize` |
| `priv/locales/client/*.toml` | Rsbuild TOML plugin | Browser `i18next` and `messageformat` |

The server catalog loader is non-recursive, so `BullX.I18n.Catalog` does not
load client keys. The browser does not fetch server catalogs.

Client keys should belong to the UI surface that renders them. Server-rendered
HEEx, channel adapter replies, and backend errors use server catalogs. React
Inertia pages use client catalogs. A key should not be duplicated across server
and client catalogs merely to share text.

The current React entry point imports the shipped client locale files directly:

```ts
import enUS from "@locales/en-US.toml"
import zhHansCN from "@locales/zh-Hans-CN.toml"
```

Adding a new browser-supported locale requires adding the TOML file and updating
`webui/src/i18n/i18n.ts` so the resource is bundled. Server-side locale support
only requires the server TOML file and a valid default-locale configuration.

## React runtime

`webui/src/i18n/i18n.ts` initializes `i18next` with:

- `initReactI18next` for React context;
- `Mf2PostProcessor` for MF2 formatting through the `messageformat` package;
- `Mf2ReactPreset` for safe basic markup support;
- `load: "currentOnly"`;
- `fallbackLng: "en-US"`;
- `initAsync: false`;
- `react.useSuspense: false`.

`webui/src/i18n/provider.tsx` exposes `BullXI18nextProvider`, and
`webui/src/app.tsx` wraps the Inertia application with that provider.

React code uses `react-i18next` directly:

```tsx
const { t } = useTranslation()

t("web.setup.title")
t("web.setup.greeting", { values: { name: "Alice" } })
```

MF2 bindings should be passed under `values`. The post-processor also accepts
non-reserved top-level i18next options for compatibility, but `values` avoids
collisions with i18next reserved options such as `lng`, `ns`, `defaultValue`, and
`postProcess`.

`Mf2PostProcessor` renders only safe markup aliases:

| MF2 alias | Rendered tag |
| --- | --- |
| `bold`, `b`, `strong` | `strong` |
| `i`, `italic`, `em` | `em` |
| `br` | `br` |
| `u` | `u` |
| `s` | `s` |
| `code` | `code` |
| `small` | `small` |

Unknown markup is dropped during rendered output. Text and fallback values are
HTML-escaped before they enter React-rendered markup.

## Validation

`mix i18n.check` is the offline catalog gate. It validates server catalogs and,
when the client catalog directory exists, client catalogs.

The check enforces:

- TOML parsing succeeds;
- every message normalizes through `BullX.I18n.Normalizer`;
- `en-US` exists as the source locale;
- every non-source locale's key set is a subset of `en-US`;
- translated values preserve the source message's MF2 variable set;
- client catalog drift is reported separately from server catalog drift.

Implementation and review should use:

```bash
MIX_ENV=test mix i18n.check
bun test webui/src/i18n
mix test test/bullx/i18n test/bullx_web/i18n test/mix/tasks/i18n
```

`bun precommit` runs the I18n check as part of the repository verification loop.

## Error and failure behavior

Server-side catalog load errors stop `BullX.I18n.Catalog` startup. This is
intentional: invalid server catalogs make user-facing text unreliable across the
Installation.

If the configured default locale does not name a loaded server catalog,
`BullX.I18n.Catalog` returns an error with the available locale list. Operators
must fix `bullx.i18n_default_locale`, `BULLX_I18N_DEFAULT_LOCALE`, or the
catalog files.

If no locale files are found, the catalog logs a warning. Runtime translation
lookups then degrade through the normal missing-key behavior.

In development, the file watcher rescans the whole server catalog directory
after TOML file changes. Rescan failures are logged and do not replace the last
successfully loaded runtime catalog.

`BullX.I18n.t/3` never raises for missing keys or MF2 format errors after the
catalog has loaded. It logs a safe diagnostic and returns a string. Callers that
cannot safely continue with degraded text must use `translate/3`.

The browser MF2 post-processor catches compile and format errors. It falls back
to an escaped source string with safe markup conversion, so page rendering does
not break because of one bad client message.

## Security, privacy, and governance

Translation catalogs are not a secret store. Catalog files must not contain API
keys, OAuth tokens, activation codes, login codes, provider raw payloads, private
profile data, or complete business records.

I18n logs may include locale ids, translation keys, and safe error summaries.
They must not include MF2 bindings when bindings can carry user content or
secrets.

I18n does not authorize side effects. Channel adapters, AIAgent Core, Command
Target, setup controllers, and other callers must complete Principal, AuthZ,
Capability, Budget, approval, or business-record checks before they render and
send localized text.

Translation output is presentation data. It must not become durable business
truth when a structured domain value is available. Durable records should store
stable ids, status atoms, Principal ids, Event ids, and other structured facts;
localized text can be rendered at the boundary where a human sees it.

## Implementation handoff

Future I18n changes should stay within the existing subsystem boundaries unless
another design doc changes the locale model.

### Goal

Maintain one cross-cutting I18n subsystem that serves server code, Phoenix
rendering, channel adapters, and React pages without reintroducing Gettext,
per-user locale state, or catalog export endpoints.

### Context pointers

- `docs/design-docs/Configuration.md` defines `BullX.Config` declarations and
  runtime resolution.
- `lib/bullx/i18n.ex` owns the server public API.
- `lib/bullx/i18n/*` owns catalog loading, normalization, and fallback.
- `lib/bullx_web/i18n/*` owns Phoenix integration.
- `webui/src/i18n/*` owns React integration.
- `priv/locales/*.toml` and `priv/locales/client/*.toml` are the catalog files.
- `lib/mix/tasks/i18n.check.ex` owns offline validation.

### Constraints

- Keep locale application-global.
- Keep server and client catalogs separate.
- Keep TOML as the only catalog format.
- Keep MF2 as the message format.
- Keep core `BullX.I18n` independent of Phoenix.
- Do not put channel, Principal, AuthZ, EventBus, AIAgent, or Workflow policy
  into I18n.
- Do not persist localized strings when a structured durable fact is available.

### Tasks

When changing I18n behavior:

1. Update the owning runtime first.
   - Owns: `lib/bullx/i18n*`, `lib/bullx_web/i18n*`, or `webui/src/i18n*`.
   - Acceptance: the changed runtime preserves the locale and catalog boundaries
     in this document.
2. Update catalog validation.
   - Owns: `lib/mix/tasks/i18n.check.ex` and its tests.
   - Acceptance: invalid catalogs fail before runtime use.
3. Update locale files.
   - Owns: `priv/locales/*.toml` and `priv/locales/client/*.toml`.
   - Acceptance: server and client key ownership remains clear.
4. Run focused verification.
   - Owns: local test execution.
   - Acceptance: `MIX_ENV=test mix i18n.check`,
     `bun test webui/src/i18n`, and focused Elixir I18n tests pass.

### Stop and ask

Stop before implementing a change that would introduce:

- per-user, per-browser, per-Principal, or per-channel locale state;
- runtime catalog export or catalog delivery through Inertia props;
- another catalog format;
- another browser or server message engine;
- a Phoenix Gettext backend;
- I18n-owned authorization, routing, or business policy.

### Done when

I18n work is done when:

- server and client catalogs validate with `MIX_ENV=test mix i18n.check`;
- server translation behavior is covered by `test/bullx/i18n*`;
- Phoenix helper behavior is covered by `test/bullx_web/i18n*`;
- React MF2 behavior is covered by `bun test webui/src/i18n`;
- `bun precommit` passes for broad changes.
