# AgenticLoop

## Summary

`BullX.AIAgent.AgenticLoop` is the runtime for Agents whose `agents.type` is
`agentic_loop`. It reads behavior configuration from the Agent Principal's
`agents.profile`, calls models through `BullX.LLM`, and stores conversation
sessions and messages in PostgreSQL.

AgenticLoop owns multi-turn conversation runtime behavior: system prompt
assembly, prompt caching, context compression, session reset, group observation
mode, and durable message storage. AgenticLoop does not define concrete tools,
skills, long-term memory schema or retrieval, subagent execution, cron Agents,
Work, Mission, Capability, Intent, Governance, Effect, or Outcome.

AgenticLoop is the conversation runtime for an Agent Principal. It is not an
LLM provider configuration extension, and sessions do not own policy. An
Installation can provide defaults, and the Agent profile can override them. The
effective behavior belongs to the Agent.

Agent conversation storage is a shared durable conversation layer for Agent
runtimes. AgenticLoop is the first user of this storage, but table names do not
carry an `agentic_loop_` prefix:

- `agent_sessions` stores conversation boundaries, source metadata, and the
  current active branch leaf.
- `agent_messages` stores the durable message tree that can be rendered into
  provider messages.

Runtime execution state must be reconstructible from the active session,
message path, provider response metadata, and logs. Only facts needed for the
conversation transcript or session lifecycle belong in durable storage.

Compression failure is generation failure. AgenticLoop must write a
`kind = error` message, preserve a safe error summary, and stop the current
generation. AgenticLoop must not insert a fallback summary marker that pretends
missing context is safe to continue from.

## Scope

This design covers:

- AgenticLoop profile configuration and Installation defaults;
- durable session and message storage;
- active branch rendering;
- slash-command session reset;
- daily session reset;
- prompt assembly and prompt caching;
- token accounting and coverage-summary compression;
- group observation mode;
- `BullX.LLM` invocation boundaries;
- failure behavior and implementation acceptance criteria.

## Non-Goals

- AgenticLoop does not define concrete tools, skills, or capability execution.
- AgenticLoop does not define long-term memory schema, retrieval, consolidation,
  or Brain internals.
- AgenticLoop does not define subagent execution, cron Agents, Work, Mission,
  Capability, Intent, Governance, Effect, or Outcome.
- AgenticLoop does not directly understand Feishu, Telegram, Discord, or other
  provider protocols.
- AgenticLoop does not let an LLM provider decide Agent policy such as group
  observation, daily reset, compression thresholds, or prompt caching.
- AgenticLoop does not add a new table for parallel tool-result groups.

## Session Model

A session is the durable boundary for one continuous conversation. Agent policy
comes from the effective Agent profile. The assembled prompt is rendered at
call time.

The table is `agent_sessions`:

| Field | Meaning |
| --- | --- |
| `id` | UUIDv7 primary key generated by BullX. |
| `agent_principal_id` | Principal id for the Agent. One Agent Principal can have multiple active sessions across different `conversation_key` values. |
| `conversation_key` | BullX-normalized key that maps source events to one conversation. |
| `current_leaf_message_id` | Current active branch leaf message. Empty for a new session with no messages. Active context rendering walks from this message through `parent_id` to the root, then reverses the path. |
| `ended_at` | `NULL` means active. Non-`NULL` means inactive. This field expresses both end time and active flag. |
| `metadata` | Non-secret JSON object. |
| `inserted_at` | Session creation time. The table does not also store `started_at`. |
| `updated_at` | Ecto timestamp. |

`metadata` stores facts that should remain with the session but do not need
dedicated columns, for example:

- source adapter, scope, thread, room, and actor summaries;
- `end_reason`, such as `new_session`, `daily_reset`, `operator_close`, or
  `agent_disabled`;
- daily reset retry count and last attempt;
- short-term generation lease information, if the runtime uses a lease to
  detect unfinished sessions.

Constraints and indexes:

- One `agent_principal_id` and `conversation_key` pair can have at most one
  active session:
  `UNIQUE (agent_principal_id, conversation_key) WHERE ended_at IS NULL`.
- `current_leaf_message_id` uses a deferrable composite foreign key to ensure
  the leaf belongs to the same session:
  `FOREIGN KEY (id, current_leaf_message_id) REFERENCES agent_messages(session_id, id) DEFERRABLE INITIALLY DEFERRED`.
  `agent_messages` must provide the matching `(session_id, id)` unique
  constraint. A `NULL` `current_leaf_message_id` means the session has no
  messages yet.
- `metadata` must be a JSON object.

The daily reset service day is derived from `inserted_at` and the effective
Agent `daily_reset_hour`. Active state is `ended_at IS NULL`. Provider policy
comes from the effective Agent profile and the `BullX.LLM` catalog.

## Message Model

A message is the durable transcript unit for Agent runtime. Prompt rendering,
compression, prompt caching, and provider calls derive from messages.

The table is `agent_messages`:

| Field | Meaning |
| --- | --- |
| `id` | UUIDv7 primary key generated by BullX. |
| `session_id` | References `agent_sessions.id`. Session deletion cascades to messages. |
| `parent_id` | References another message in the same table. `NULL` means the root message inside a session. Multiple children under the same parent express branches. |
| `role` | Renderable provider role: `system`, `user`, `assistant`, or `tool`. Hidden `command` and `error` messages still store the nearest causal role. Commands use `user`; generation or compression failures use `assistant`. |
| `kind` | BullX message kind: `normal`, `summary`, `command`, or `error`. |
| `status` | `complete` or `generating`. `generating` is only for crash recovery. Failed facts are written as `kind = error` messages. |
| `content` | JSONB content parts aligned with provider content blocks. It can store text, image, tool use, tool result, and related blocks. |
| `covers_range` | JSONB coverage marker used only by `kind = summary`, shaped as `{"from_id": "<uuid>", "to_id": "<uuid>"}`. It means the summary covers the closed interval on the same active path from `from_id` to `to_id`. |
| `metadata` | Non-secret JSON object. |
| `inserted_at` | Write time. UUIDv7 and `inserted_at` support debugging order. |
| `updated_at` | Ecto timestamp. Status updates from `generating` to `complete` change this field. |

`metadata` stores provider and source details, for example:

- external source message id, route decision id, and actor summary;
- assistant tool use block, tool `tool_call_id`, and tool name;
- provider, model, request id, usage, finish reason, and reasoning metadata;
- token count and `token_count_source = provider | estimated`;
- `input_mode = direct | mentioned_group | observed_group`;
- parallel tool result group metadata, such as `tool_call_group_id` and
  `parallel = true`;
- compression count, summary source hints, and cache marker debug info;
- command alias, such as `/new` or `/新会话`.

Constraints and indexes:

- `role` is limited to `system`, `user`, `assistant`, or `tool`.
- `kind` is limited to `normal`, `summary`, `command`, or `error`.
- `status` is limited to `complete` or `generating`.
- `covers_range` is allowed only on summary messages:
  `(kind = 'summary' AND covers_range IS NOT NULL) OR (kind <> 'summary' AND covers_range IS NULL)`.
- `id <> parent_id` prevents direct self-cycles. More complex cycles are
  avoided by application-level append-only write rules.
- `agent_messages (session_id, inserted_at)` supports session debugging lists
  and active path queries.
- `agent_messages (parent_id)` supports branch-switching UI queries for sibling
  messages.
- `agent_messages (session_id, id)` is a unique constraint that supports the
  `agent_sessions.current_leaf_message_id` deferrable composite foreign key.

The application must validate that `covers_range.from_id` and
`covers_range.to_id` belong to the same session, appear on the same active
path, and that the summary message is a descendant after the covered range.
PostgreSQL checks only enforce field shape and the relation between `kind` and
`covers_range`. AgenticLoop validates ancestry in the same transaction.

Message ordering is expressed by the parent tree and `inserted_at`. Token
statistics belong in message metadata. Compression changes prompt rendering
through coverage summaries while retaining raw transcript rows.

Regenerate and edit-and-resend create sibling branches that share the original
message's `parent_id`. Branch switching updates only
`agent_sessions.current_leaf_message_id`.

Parallel tool calls are expressed as one assistant message containing multiple
tool use content blocks. The corresponding tool results are written as one
`role = tool`, `kind = normal` message whose `content` array contains all
`tool_result` blocks. This tool message's `parent_id` points to the assistant
message that initiated tool use. The next assistant message then uses the tool
message as its parent. Parallelism is stored in tool message metadata, such as
`tool_call_group_id` and `parallel = true`.

Active context consists of one canonical path, coverage summaries, and provider
rendering rules. The canonical path is built by walking from
`agent_sessions.current_leaf_message_id` through `agent_messages.parent_id` to
the root, then reversing the path. The renderer does not render
`kind = command` or `kind = error` messages. When the renderer encounters a
`kind = summary` message, it records the message's `covers_range`, inserts the
summary content at the covered range's position, and skips the summary row at
its own storage position. Tool result messages are already on the linear path,
so the renderer does not need sibling merge rules. The renderer does not scan
all session messages and does not assemble messages outside the active branch
by time window.

## Conversation Key

`conversation_key` is a BullX-internal key. It is not an external platform id.
The key is determined by the Agent, source, and isolation policy:

- `agent_principal_id`;
- source kind and adapter;
- DM, group, channel, thread, and similar scope;
- thread or topic id;
- group isolation mode;
- external actor id, when Agent configuration requires per-actor isolation
  inside group chat.

When `listen_all_group_messages = true`, unmentioned group messages use the
same group/thread conversation key. This lets the Agent see room context instead
of turning each observation into an isolated event.

`conversation_key` stores only the normalized key. The parts used to build that
key are Router process data. If those parts need to be queried or debugged,
AgenticLoop stores them in metadata.

Ingress deduplicates by external message id. If a database fallback is needed,
the external id is stored in message metadata and constrained with a JSON-path
partial unique index.

   `current_leaf_message_id` to the summary message. If compression fails,
   AgenticLoop writes a `kind = error` message and stops the current generation.
10. AgenticLoop assembles system prompt and provider messages from the active
    context after coverage summaries have been applied.
11. AgenticLoop applies prompt cache markers when the provider supports them and
    the Agent enables them.
12. AgenticLoop calls `main_llm` through `BullX.LLM`.
13. AgenticLoop appends an assistant message or error message and moves the
    session leaf to the new message.
14. AgenticLoop Runtime, acting for the Agent Principal, decides whether to
    construct an external reply `Delivery` for ordinary conversational replies.
    If the reply is a governed Effect, Governance or Effect approval must happen
    before submission. Gateway only accepts and transports an already
    constructed and allowed `Delivery` through `BullX.Gateway.deliver/1`.

Generation inside the same session must be serialized. Serialization can use a
GenServer, Oban unique job, PostgreSQL lock, advisory lock, or runtime lease.
This design does not preselect an OTP topology. Whatever implementation is
chosen, PostgreSQL sessions and messages must be sufficient to rebuild
conversation state after process restart.

## Slash Commands And I18n

`/new` and `/新会话` are built-in aliases. Both are valid in every locale.

Slash command rules:

- Command detection happens before the LLM call.
- A command message is a hidden transcript control record. AgenticLoop
  persists it to the affected session as `role = user`, `kind = command`,
  stores the raw command text in `content`, and stores the matched alias in
  `metadata.command_alias`. The prompt renderer does not render any
  `kind = command` message.
- `/new` and `/新会话` close the current active session. AgenticLoop first
  appends the command message and updates the old session leaf, then sets
  `ended_at = now()` and writes `metadata.end_reason = new_session`.
- The new session inherits the same `agent_principal_id`, `conversation_key`,
  and source metadata. Its initial `current_leaf_message_id` is `NULL`.
- `/new` does not write session lineage. It means the user explicitly starts a
  new sibling session, not a child continuation.

Additional language aliases should be added through i18n resources or an
AgenticLoop command registry, but `/new` is a non-removable default alias.

## Daily Reset

Daily reset runs at the effective Agent `daily_reset_hour`, defaulting to
`04:00`. Daily reset is session hygiene. A reset worker computes it from session
creation time and the effective Agent profile.

Rules:

- The reset worker scans sessions where `ended_at IS NULL` and uses session
  `inserted_at`, current time, and the effective Agent `daily_reset_hour` to
  decide whether the service day has changed.
- If the session has no unfinished generation, the worker closes the old session
  and creates a new active session. The old session gets
  `metadata.end_reason = daily_reset`.
- If the session is generating, the worker does not close it. The worker retries
  after the effective Agent `daily_reset_retry_minutes`, defaulting to 30
  minutes.
- Retries continue until reset succeeds.

Unfinished generation detection is a runtime implementation detail, but daily
reset must be able to decide it reliably. Valid implementations include
session-scoped GenServer state, an Oban running job, PostgreSQL advisory lock,
or a short-term lease in session metadata. If a lease is used, the lease must
have an expiration time so a crash cannot permanently block reset.
