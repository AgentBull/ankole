# EventBus streaming output

TargetSession output streams use Redis as a weak runtime buffer for live
follow and reconnect/resume behavior. The buffer stores bounded output chunks in
a Redis List, stream metadata in a Redis Hash, and live notifications as
Pub/Sub pointers. The buffer is runtime output state only: it is not Event
routing, not the inbound TargetSession side channel, not a business transcript,
and not audit or durable replay truth.

## Scope

This design covers the EventBus output stream surface:

- TargetSession output stream ownership and lifecycle.
- Redis metadata, chunk, and Pub/Sub keys.
- `create_stream/3`, `append_chunk/2`, `finish_stream/3`, `resume_stream/2`,
  and `follow_stream/3` API contracts.
- Stream offset conventions, TTL, retention, and configuration defaults.
- Live follow behavior, pointer notifications, and resume race handling.
- Producer crash behavior and client disconnect semantics.
- Gateway live stream transport consumption.
- Stream telemetry, logging, and implementation tests.

`Core.md`, `Matcher.md`, and `Persistence.md` own Event acceptance, route
matching, TargetSession resolution, the inbound TargetSession side channel, Oban
session-loop behavior, and core PostgreSQL runtime schema. This document only
references `target_session_id` and `target_session_entry_id` as stream ownership
anchors created by the core EventBus and TargetSession flow.

## Goals

- Let AIAgent and Workflow Targets stream output while a TargetSession is
  running.
- Let clients reconnect and resume from the last emitted chunk offset while the
  Redis runtime buffer still exists.
- Let multiple live consumers follow the same stream without making Pub/Sub the
  content carrier.
- Keep browser or client disconnect separate from producer stop or cancel.
- Keep generated stream chunks out of EventBus business truth. Targets or
  Conversation layers persist business transcript when transcript persistence is
  required.

## Non-goals

- PostgreSQL stream tables. EventBus streaming output does not add logged,
  unlogged, or temporary PostgreSQL tables for stream chunks.
- Event acceptance, route matching, TargetSession resolver behavior, inbound
  side-channel storage, or Oban TargetSession loop design.
- Business transcript persistence. Stream chunks are not Conversation or Message
  records.
- Audit, compliance replay, or durable artifact storage. Business layers must
  write those facts explicitly.
- Explicit stop, abort, or cancel semantics for streaming output. Resume
  semantics do not interpret client disconnect as stop or cancel.
- Gateway routing policy. Gateway may expose a live transport, but Gateway does
  not evaluate Event Routing Rules, create TargetSessions, or inspect Target
  internals.

## Runtime ownership

A TargetSession may create one or more output streams while handling a
side-channel entry. Each stream belongs to a `target_session_id` and may also
belong to a `target_session_entry_id`. A stream without a specific entry stores
an empty `target_session_entry_id` field in Redis.

The stream producer is the TargetSession job or Target code currently writing
that stream. The producer appends chunks and finishes the stream with a
terminal status. Stream consumers never write chunks and never own producer
lifecycle. A consumer disconnect only closes that consumer's live follow path;
the producer continues unless a separate, explicit business or control path
stops it.

Redis stream state is weak runtime state with TTL and retention. Missing or
expired Redis state means the stream cannot be resumed from EventBus. EventBus
must return `:unavailable` or `:no_content`; it must not fabricate missing
chunks, reconstruct output from logs, or infer transcript from Target state.

## Redis key model

Each output stream uses one metadata hash, one chunk list, and one Pub/Sub
channel. `stream_id` is an opaque UUIDv7 string generated by BullX code.

### Metadata hash

```text
bullx:stream:{stream_id}:meta
```

The metadata hash stores stream lifecycle and ownership fields:

| Field | Meaning |
| --- | --- |
| `status` | `open`, `completed`, `failed`, `interrupted`, or `expired`. |
| `next_offset` | Next chunk offset to allocate. The first chunk uses offset `0`. |
| `target_session_id` | UUID of the owning TargetSession. |
| `target_session_entry_id` | UUID of the owning entry, or empty when the stream is session-scoped. |
| `created_at` | Unix timestamp in milliseconds. |
| `updated_at` | Unix timestamp in milliseconds. |
| `expires_at` | Unix timestamp in milliseconds when the runtime buffer should be considered expired. |
| `terminal_reason` | Optional safe terminal code or short diagnostic text. |

`terminal_reason` is a short diagnostic code or summary built by the producer.
It is not a place to copy Event payloads, stream chunks, credentials, or raw
provider payloads.

### Chunk list

```text
bullx:stream:{stream_id}:chunks
```

The chunk list is a Redis List. Each element stores the exact text chunk or SSE
fragment emitted by the producer. The Redis List index is the stream offset.
Implementations must append chunks sequentially and must not leave intentional
offset gaps.

The list is the replay truth for the runtime stream. Pub/Sub notifications point
to offsets in this list; Pub/Sub does not carry chunk text.

### Pub/Sub channel

```text
bullx:stream:{stream_id}:pub
```

The Pub/Sub channel serves live stream consumers. Notification payloads are
pointers and may include only:

- `stream_id`
- `offset` for a chunk notification
- terminal `status` for a terminal notification
- safe diagnostics such as a diagnostic code

Notification payloads are constructed only from the pointer fields above.

Redis Pub/Sub is at-most-once. Losing a Pub/Sub notification must not lose
content. Consumers recover by reading missing offsets from the Redis List.

## Configuration and retention

Stream runtime bounds are configuration values with implementation-owned
defaults. This design requires the defaults to exist and be tested, but it does
not fix product-specific numeric values.

| Config | Meaning |
| --- | --- |
| `stream_retention_seconds` | TTL for metadata and chunks after creation or the latest append/finish refresh. |
| `max_stream_chunk_bytes` | Maximum UTF-8 byte size for one chunk. |

All existing stream keys receive TTLs. Metadata receives a TTL during stream
creation. The chunks key receives its TTL on first append, then open streams
refresh metadata and chunk TTLs on later appends. Finished streams keep
metadata and chunks until retention expires.

Redis cleanup is not transcript cleanup. When stream keys expire, EventBus loses
runtime resume capability for that stream. If output must survive retention, the
Target or business layer must persist it as Conversation, Message, Artifact, or
another business fact before the Redis buffer expires.

## API contracts

The stream buffer exposes five minimal functions. The module boundary may live
under the EventBus implementation, but these shapes are the public design
contract.

```elixir
create_stream(target_session_id, target_session_entry_id, opts) ::
  {:ok, stream_id}
  | {:error, :redis_unavailable | :invalid_opts}

append_chunk(stream_id, chunk_text) ::
  {:ok, offset}
  | {:error, :stream_not_found | :stream_not_open | :redis_unavailable | :chunk_too_large}

finish_stream(stream_id, status, terminal_reason) ::
  :ok
  | {:error, :stream_not_found | :redis_unavailable}

resume_stream(stream_id, after_offset) ::
  {:ok,
   %{
     status: :open | :completed | :failed | :interrupted,
     chunks: [%{offset: non_neg_integer(), chunk: String.t()}],
     follow?: boolean()
   }}
  | {:error, :unavailable | :no_content | :redis_unavailable}

follow_stream(stream_id, after_offset, consumer) ::
  :ok
  | {:error, :unavailable | :redis_unavailable}
```

`finish_stream/3` accepts `:completed`, `:failed`, or `:interrupted` as terminal
statuses. `:expired` is a cleanup state, not a successful `finish_stream/3`
input and not an `{:ok, ...}` resume status.

`opts` may carry safe producer diagnostics and stream formatting hints. `opts`
must not override `target_session_id`, `target_session_entry_id`, retention, or
chunk-size limits in a way that weakens the configured runtime bounds.

## Offset convention

Stream offsets start at `0`. `after_offset` means "the last offset the consumer
successfully emitted." A consumer that wants to resume from the beginning passes
`after_offset = -1` or `nil`; the implementation normalizes `nil` to `-1`.

`resume_stream/2` returns chunks whose offsets are greater than `after_offset`.
Consumers should treat offsets as idempotency keys and ignore duplicate offsets
when race handling causes a replayed chunk to arrive through both buffered
resume and live follow.

`follow_stream/3` uses the same `after_offset` convention. It invokes
`consumer` for each chunk or terminal event after the last emitted offset.

## Stream operations

### Create stream

`create_stream/3` creates the metadata hash, initializes `next_offset = 0`,
stores TargetSession ownership fields, applies a TTL to metadata, and returns
`stream_id`. The chunks key does not exist before the first `RPUSH`; it
receives TTL on first append and every later append.

`create_stream/3` fails with `{:error, :redis_unavailable}` when Redis cannot
complete the required writes. It fails with `{:error, :invalid_opts}` when the
provided options contain unsupported or unsafe values.

### Append chunk

`append_chunk/2` must be atomic. The preferred implementation is one Redis Lua
script that:

1. Checks that metadata exists and `status == open`.
2. Rejects `chunk_text` when its UTF-8 byte size exceeds
   `max_stream_chunk_bytes`.
3. Reads `next_offset` as the allocated `offset`.
4. Increments `next_offset`.
5. Appends `chunk_text` to the chunk list with `RPUSH`.
6. Updates `updated_at` and refreshes metadata and chunk TTLs.
7. Publishes a pointer notification with `stream_id` and `offset`.
8. Returns `offset`.

The notification is published after the chunk is in the Redis List. A consumer
that receives the notification must read chunk text from the list by offset.

### Finish stream

`finish_stream/3` changes an open stream to `completed`, `failed`, or
`interrupted`, writes a safe `terminal_reason` when provided, updates
`updated_at`, publishes a terminal pointer notification, and preserves metadata
and chunks until retention expires.

Finishing a missing stream returns `{:error, :stream_not_found}`. Finishing a
non-open stream should be idempotent when the existing terminal status and safe
terminal reason match the requested terminal state; conflicting terminal writes
must not rewrite history.

### Resume stream

`resume_stream/2` reads stream metadata and chunk list state from Redis:

- Missing or expired metadata returns `{:error, :unavailable}` or
  `{:error, :no_content}`.
- `open` streams return buffered chunks with `offset > after_offset` and
  `follow?: true`.
- `completed`, `failed`, or `interrupted` streams return buffered chunks with
  `offset > after_offset` and `follow?: false`.
- `expired` streams return `{:error, :unavailable}` or `{:error, :no_content}`.

For an `open` stream, the `resume_stream/2` and `follow_stream/3` pair must
handle the catch-up/live-follow race. `resume_stream/2` returns buffered chunks
first. `follow_stream/3` subscribes to the Pub/Sub channel with the latest
emitted offset and reads any gap from the Redis List after the subscription is
established. Duplicate offsets may be discarded by the consumer. Pub/Sub loss
cannot drop content because the Redis List remains the runtime replay source
until retention expires.

### Follow stream

`follow_stream/3` enters live follow for an open stream. It subscribes to the
Pub/Sub pointer channel, reads chunk text from the Redis List by offset, repairs
gaps by reading ranges from the list, and invokes `consumer` for each chunk and
terminal event.

`follow_stream/3` must not pass Pub/Sub payloads through as chunk content.
Pub/Sub messages are pointers only. If metadata is missing, expired, or no
longer followable, `follow_stream/3` returns `{:error, :unavailable}`. If Redis
cannot subscribe or read, it returns `{:error, :redis_unavailable}`.

## Live follow consumers

A live follow consumer keeps its own last emitted offset. When a Pub/Sub chunk
notification arrives, the consumer reads that offset, or a range from
`last_emitted_offset + 1`, from the chunk list. If the consumer detects a gap, it
reads the missing range before emitting newer chunks.

Multiple consumers may follow the same stream. Consumers do not extend producer
lifetime and do not keep the stream alive beyond TTL. If all consumers
disconnect, the producer continues to run and append chunks until it finishes,
crashes, or is stopped by a separate control path outside resume semantics.

## Failure behavior

### Redis unavailable

When Redis is unavailable, create, append, finish, and resume return
`:redis_unavailable` where the API shape exposes that error. A resume attempt
may also return `:unavailable` when the caller-facing contract should not expose
the difference between missing runtime state and an unavailable stream.

### Missing or expired state

Missing or expired metadata or chunks mean EventBus can no longer resume the
runtime stream. EventBus returns `:unavailable` or `:no_content`. EventBus must
not reconstruct chunks from application logs, Target internals, business
records, or Conversation state.

### Pub/Sub loss

Pub/Sub notifications are hints. A lost notification only delays live delivery.
The consumer repairs by reading from the Redis List using its last emitted
offset.

### Producer crash

If the producer crashes before calling `finish_stream/3`, Redis may still show
the stream as `open` until retention expires. EventBus does not add a producer
lease or liveness detector for this edge case. A TargetSession retry, explicit
Target failure path, or business layer may create a new stream id for the
retried attempt.

EventBus must never fill missing offsets with synthetic chunks.

### Client disconnect

Client or browser disconnect closes only that live consumer path. It is not
producer abort, stop, or cancel. A later client may resume from its last emitted
offset while the Redis buffer remains available. Explicit stop or cancel must be
modeled as a separate Event or API outside this stream resume contract.

## Gateway live transport

Gateway and adapters remain transport-only. If a Gateway exposes Server-Sent
Events, WebSocket streaming, or a provider-specific live response transport, it
consumes the TargetSession output stream through the stream buffer APIs:

1. Call `resume_stream/2` with the client-provided `stream_id` and
   `after_offset`.
2. Emit buffered chunks in offset order.
3. When `follow?` is true, call `follow_stream/3` with the latest emitted
   offset and emit chunks or terminal events from that callback path.
4. Stop only the consumer connection when the client disconnects.

Gateway must not evaluate Event Routing Rules, create TargetSessions, append
inbound side-channel entries, create Oban jobs, infer Target internals, or write
Conversation transcripts from stream chunks.

## Telemetry and logging

Stream telemetry uses EventBus namespaced events:

- `[:bullx, :event_bus, :stream, :created]`
- `[:bullx, :event_bus, :stream, :chunk_appended]`
- `[:bullx, :event_bus, :stream, :completed]`
- `[:bullx, :event_bus, :stream, :interrupted]`
- `[:bullx, :event_bus, :stream, :expired]`
- `[:bullx, :event_bus, :stream, :resume_requested]`
- `[:bullx, :event_bus, :stream, :resume_unavailable]`

Allowed metadata includes:

- `stream_id`
- `target_session_id`
- `target_session_entry_id`
- Event Routing Rule id when already available from Core-owned context
- target type
- stream status
- offset
- chunk byte size
- source hash
- diagnostic code

Telemetry, logs, Pub/Sub notifications, and `terminal_reason` are constructed
only from this allowlist. Runtime code must not pass stream chunks, Event
payloads, credentials, tokens, or provider raw payloads into metadata builders.

Measurements stay small. Useful measurements include create duration, append
duration, resume duration, returned chunk count, and chunk byte size. The implementation should
not add a broad metrics taxonomy before the implementation has real operational
pressure.

## Security and privacy

Redis stream chunks may contain private messages, generated assistant text,
customer names, file references, and other domain-sensitive content. Operators
must treat the Redis runtime store as sensitive runtime infrastructure. The
bounded TTL limits exposure but does not make stream content safe to log or
publish.

Stream ids are bearer-style references to runtime output while the Redis buffer
exists. Gateway and API layers must authorize access to the TargetSession output
before returning chunks to a Principal or external client. This document does not
change Principal, AuthZ, Governance, Capability, or Target-owned policy
decisions.

## Deliberate tradeoffs

Redis provides reconnect and live follow for short-lived runtime output, not
durable transcript guarantees. This keeps EventBus streaming simple and avoids
creating a second business-message store inside EventBus.

Pub/Sub carries only pointers. Consumers do one extra Redis List read, but lost
Pub/Sub messages do not lose content while the list remains within retention.

Producer crash is handled by the weak runtime boundary instead of a producer
lease. The stream may remain open until TTL when the producer dies without a
terminal write. This leaves a rare edge case to TargetSession retry, explicit
failure handling, or operator diagnosis, and avoids a second liveness mechanism
inside EventBus streaming.

Client disconnect does not cancel the producer. This matches the requirement
that browser reconnect and live streaming remain transport concerns, while stop
or cancel remains an explicit control behavior outside resume semantics.

## Implementation handoff

### Goal

Implement Redis-backed TargetSession output streams for EventBus, with the
API contracts, key model, offset behavior, failure semantics, Gateway
consumption boundary, and tests defined in this document.

### Context pointers

- `docs/Architecture.md` defines the EventBus, TargetSession, Target, and weak
  Redis runtime buffer invariants.
- `Core.md`, `Matcher.md`, and `Persistence.md` own Event acceptance, route
  matching, TargetSession resolution, inbound TargetSession side-channel
  behavior, Oban session-loop behavior, and core PostgreSQL runtime records.
- `AGENTS.md` defines repository rules, including BullX-side UUIDv7 generation,
  design-doc-first workflow, and verification expectations.

### Constraints

- Use Redis only for TargetSession output stream buffering and live follow.
- Do not use Redis as the inbound Event side channel, TargetSession wake
  mechanism, Oban replacement, business transcript, audit log, or durable replay
  store.
- Do not add PostgreSQL stream chunk tables for output streaming.
- Do not put chunk text in Pub/Sub notifications, telemetry, logs, or terminal
  diagnostics.
- Do not treat client disconnect as producer stop or cancel.
- Do not fabricate missing chunks after Redis expiry, Redis loss, producer
  crash, or offset gaps.
- Do not make Gateway own routing policy, TargetSession creation, Target
  internals, or Conversation persistence.

### Tasks

1. Define stream configuration.
   - Owns: EventBus stream configuration declarations and tests.
   - Depends on: None.
   - Acceptance: `stream_retention_seconds` and `max_stream_chunk_bytes` exist
     with tested defaults and validation rejects unsafe values.

2. Implement Redis key helpers.
   - Owns: EventBus stream buffer key construction.
   - Depends on: Task 1.
   - Acceptance: Metadata, chunk list, and Pub/Sub keys match the key model and
     isolate streams by `stream_id`.

3. Implement `create_stream/3`.
   - Owns: Stream buffer API and create tests.
   - Depends on: Tasks 1 and 2.
   - Acceptance: Creation writes metadata, initializes `next_offset = 0`, stores
     ownership fields, applies metadata TTL, leaves the chunks key to receive
     TTL on first append, returns `{:ok, stream_id}`, and reports Redis or
     option errors with the specified return shapes.

4. Implement atomic `append_chunk/2`.
   - Owns: Redis Lua script or equivalent atomic append path and tests.
   - Depends on: Task 3.
   - Acceptance: Append checks `open`, enforces chunk byte size, allocates
     sequential offsets from `0`, writes the chunk before publishing a pointer,
     refreshes TTLs, never publishes chunk text, and returns the allocated
     offset.

5. Implement `finish_stream/3`.
   - Owns: Terminal stream state, terminal notification, and tests.
   - Depends on: Task 3.
   - Acceptance: Finish records `completed`, `failed`, or `interrupted`, stores
     only safe terminal diagnostics, publishes a terminal pointer notification,
     preserves metadata and chunks for retention, and handles matching repeated
     terminal writes idempotently.

6. Implement `resume_stream/2` and live follow race protection.
   - Owns: Resume API, buffered replay, Pub/Sub subscription flow, and tests.
   - Depends on: Tasks 3, 4, and 5.
   - Acceptance: Resume normalizes `nil` to `-1`, returns chunks with
     `offset > after_offset`, sets `follow?` correctly, handles missing and
     expired state, handles terminal states, and prevents subscribe/replay races
     from losing chunks.

7. Implement `follow_stream/3`.
   - Owns: Pub/Sub subscription, pointer handling, Redis List reads, gap repair,
     consumer callback integration, and tests.
   - Depends on: Tasks 3, 4, 5, and 6.
   - Acceptance: Follow subscribes to the pointer channel, reads chunk text from
     the Redis List by offset, repairs gaps by range reads, invokes the consumer
     for chunks and terminal events, never treats Pub/Sub messages as chunk
     carriers, and returns `:unavailable` or `:redis_unavailable` as specified.

8. Implement Gateway live stream consumption tests.
   - Owns: Fake Gateway or adapter tests around the stream API.
   - Depends on: Tasks 6 and 7.
   - Acceptance: Gateway consumes buffered chunks and pointer notifications,
     reads chunk text from Redis List by offset, keeps disconnect scoped to the
     client connection, and does not take over routing, TargetSession creation,
     or Conversation persistence.

9. Add telemetry and logging tests.
   - Owns: Stream telemetry assertions and log-safety checks.
   - Depends on: Tasks 3 through 7.
   - Acceptance: Stream telemetry uses the namespaced events in this document
     and is built only from the allowlisted metadata fields.
